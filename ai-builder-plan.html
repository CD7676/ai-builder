02

How the dashboard works

Static HTML on the front, a Notion database on the back, and a serverless function in between. Single-page Kanban; everything lives in one source of truth.

Layer-001 / Front end Shipped

A custom Kanban with pointer-driven drag/drop and FLIP animations — no framework.

Stack
Plain HTML, CSS variables, vanilla JS. fetch hydrates from /api/roadmap; render is one pass over the response.
Interaction
Pointer events instead of HTML5 drag, so the same code works on touch and mouse. Position changes use the FLIP pattern (First, Last, Invert, Play) with the Web Animations API to spring cards into their new slots. Click advances a card right; right-click moves it left. Each move PATCHes Notion in the background.
Layer-002 / API Shipped

A serverless function that turns a Notion database into a JSON Kanban feed.

Endpoint
GET /api/roadmap queries the Notion data source via @notionhq/client, paginates the results, and maps each page to a flat card object. PATCH /api/roadmap updates a card's Status select by ID.
Filtering
The function reads a PUBLIC_ONLY environment variable. When set, it filters out any row whose Public checkbox is false — so the same code can power a public showcase and a private mirror without divergent branches.
Layer-003 / Data Shipped

Notion is the database, the editor, and the spec viewer in one.

Schema
A single database with Name, Description, Project spec, Priority, Status, Public, Submitted by, Submitted at. Schema changes go through the Notion MCP, so they're scriptable.
Why
Editing a card means typing into Notion, not opening a CMS or pushing code. The Public checkbox is the one bit of access control: flip it on, the card shows up here within the next request.
Layer-004 / Deploy Shipped

GitHub push → Vercel build → live in under a minute.

Pipeline
Vercel's Git integration watches main. Every push triggers a build of the static files plus the serverless function. NOTION_TOKEN and NOTION_DATABASE_ID live in Vercel env vars, never in the repo.
Two deployments, one repo pattern
The same code runs as a public showcase and as a private full version, separated only by env vars and deployment protection. No code forks; the difference is a single boolean.
